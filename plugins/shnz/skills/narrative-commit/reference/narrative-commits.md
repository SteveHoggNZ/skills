# Narrative commit messages — full reference

Extended guidance for the narrative-commit skill. The canonical procedure lives in [../core.md](../core.md); this file is the "load on demand when a judgement call is unclear" companion.

## Philosophy

Commits should read like **chapters in a book** — each one tells a clear story about one logical change. The message focuses on **what changed and why**, not how it was produced.

Key principles:

- **One logical change per commit.** Schema changes separate from UI updates.
- **Clear, descriptive messages.** "Add user authentication" beats "Update files".
- **Grouped by functionality.** Related changes go together (all new types in one commit, all new components in another).
- **Build incrementally.** Each commit ideally leaves the codebase in a working state.

## Format

Conventional Commits:

```
<type>(<optional scope>): <description>

<optional body>

<optional footer>
```

### Types

| Type | When to use | Example |
|------|-------------|---------|
| `feat` | New feature or capability | `feat(auth): add JWT token validation` |
| `fix` | Bug fix | `fix(channels): prevent duplicate message rendering` |
| `refactor` | Code restructuring without behaviour change | `refactor(api): extract validation into middleware` |
| `docs` | Documentation only | `docs: update API endpoint examples` |
| `test` | Adding or updating tests | `test(messages): add streaming message tests` |
| `chore` | Build, config, dependencies | `chore: update dependencies` |
| `style` | Formatting, whitespace (not CSS) | `style: format with prettier` |
| `perf` | Performance improvement | `perf(db): add index on channel_id` |

### Scope (optional)

Scope names the area of the codebase affected — `(auth)`, `(channels)`, `(messages)`, `(ui)`, `(api)`, `(db)`, etc. Use the project's existing scope vocabulary if there is one; otherwise pick names that match directory/module structure.

### Description

- Start lowercase (unless a proper noun).
- No trailing period.
- Present tense, imperative mood: "add", not "added" or "adds".
- Be specific: "add mention autocomplete" beats "update UI".

Good:

```
feat(mentions): add autocomplete for @mentions
fix(scroll): preserve position when navigating channels
refactor(messages): extract message reducer logic
```

Bad:

```
Updated files                    — too vague
Fixes stuff                      — not descriptive
Added feature.                   — trailing period, past tense
feat: Implement Authentication   — capitalised description
```

## What NOT to include

**No AI / agent attribution.** Ever. Not `🤖 Generated with [Claude Code]`, not `Co-Authored-By: Claude …`, not any "generated by" metadata. Commits should read as if a human wrote them; the focus is the change, not the authorship method. If the harness's default behaviour wants to append attribution, the skill overrides it.

## When to add a body

Add a body when the **why** isn't obvious from the description:

```
fix(channels): prevent scroll jump on new messages

The scroll container recalculated height on every message,
causing the viewport to jump. Now we measure distance from
bottom before and after the update and restore it.
```

Add a body for:

- Complex bug fixes that need explanation.
- Non-obvious architectural decisions.
- Breaking changes (include a `BREAKING CHANGE:` footer).
- Changes that affect multiple systems.

Skip the body for:

- Self-explanatory changes ("add missing import").
- Simple refactors ("rename variable").
- Trivial fixes ("fix typo").

## Grouping strategies

### By layer (default for full-stack changes)

1. Database / schema.
2. Backend / API.
3. Types / interfaces.
4. UI components.
5. Styling.
6. Tests.
7. Documentation.

Example sequence:

```
feat(db): add notifications table and indexes
feat(api): add notification endpoints
feat(types): add notification TypeScript types
feat(ui): add NotificationsPane component
feat(ui): add notification badge to AppRail
test(notifications): add notification flow tests
docs: update API documentation with notification endpoints
```

### By feature (for isolated feature work)

Group all related changes for one feature, even across layers:

```
feat(mentions): add mention entity tracking
feat(mentions): add autocomplete UI component
feat(mentions): integrate mention detection in composer
test(mentions): add mention parsing tests
```

### By file type (sparingly)

Only when the changes are genuinely independent:

```
refactor: extract utility functions
refactor: update type definitions
refactor: reorganise component structure
```

If you're reaching for this strategy, it usually means two features got tangled. Prefer splitting by feature first.

## Common patterns

### Feature addition

```
feat(auth): add password reset flow
feat(channels): add channel archiving
feat(messages): add message editing capability
```

### Bug fixes

```
fix(scroll): preserve position on channel navigation
fix(mentions): handle mentions in thread replies
fix(api): prevent duplicate message delivery
```

### Refactoring

```
refactor(messages): extract reducer logic into separate file
refactor(api): consolidate error handling middleware
refactor(ui): unify message component variants
```

### Database changes

```
feat(db): add user_preferences table
fix(db): add missing foreign key constraint on messages
perf(db): add composite index on (channel_id, sent_at)
```

## Multi-file commits

It's fine — often preferable — to have multiple files in one commit if they're part of the same logical change:

**Good:**

```
feat(notifications): add notification system

- Add NotificationsPane.tsx component
- Add NotificationsContext.tsx for state management
- Update AppRail.tsx to show notification badge
- Add notification types to types.ts
```

**Bad:**

```
feat: updates

- Add notifications
- Fix scroll bug
- Update API endpoints
- Refactor styling
```

The bad example mixes unrelated changes — each bullet is its own commit.

## Worked examples

### Good narrative sequence

```
1. feat(db): add thread_subscriptions table for notification tracking
2. feat(api): add endpoint to subscribe/unsubscribe from threads
3. feat(types): add ThreadSubscription and NotificationPreference types
4. feat(ui): add NotificationsPane component with subscription list
5. feat(ui): integrate notification preferences into settings
6. fix(notifications): prevent duplicate notifications for same event
7. test(notifications): add subscription flow integration tests
8. docs: document notification system architecture
```

Clear progression: database → backend → types → UI → fixes → tests → docs. Each commit is focused and tells part of the story.

### Bad commit sequence

```
1. feat: add stuff
2. fix: more changes
3. Update files
4. WIP
5. Final changes
6. Actually final
```

Vague, no structure, no logical grouping. Impossible to understand without reading every diff.

## Tips

1. **Use `git add -p`** for surgical commits — stage specific hunks of a file.
2. **Review staged changes** before committing: `git diff --cached`.
3. **Write the message first.** If you can't describe the change clearly, it's probably not focused enough — split it.
4. **Think about future readers** (including yourself in six months).
5. **Ask "why would someone care about this commit?"** Your message should answer that.

## Integration with other workflows

### Active development

You don't need perfect commits while exploring. Use WIP commits freely:

```
WIP: experimenting with mention detection
WIP: trying different scroll approach
```

Clean them up before merging.

### Before merging / opening a PR

Run the narrative-commit skill over the working tree (or use `git rebase -i` for pushed history, with care) to produce a readable sequence.

### Quick fixes

Even one-line commits follow the format:

```
fix(ui): correct alignment of send button
chore: update playwright to 1.40.0
docs: fix typo in README
```
